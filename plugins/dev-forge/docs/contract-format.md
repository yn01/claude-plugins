# Sprint Contract Format

## SQLite Schema

```sql
CREATE TABLE contracts (
  id TEXT PRIMARY KEY,            -- e.g., CONTRACT-20260407-143000
  task TEXT NOT NULL,             -- Human-readable task description
  team_lead TEXT NOT NULL,        -- Agent ID of the owning team lead
  status TEXT NOT NULL DEFAULT 'active',  -- active | completed | blocked
  criteria TEXT NOT NULL,         -- JSON array of acceptance criterion strings
  created_at TEXT NOT NULL,       -- ISO 8601 datetime
  completed_at TEXT,              -- ISO 8601 datetime (null until completion)
  notes TEXT                      -- Optional completion notes or escalation info
);
```

## Contract ID Format

`CONTRACT-YYYYMMDD-HHMMSS` (e.g., `CONTRACT-20260407-143022`)

Generated by: `CONTRACT-$(date +%Y%m%d-%H%M%S)`

## Status Transitions

```
active ──── PASS evaluation ──► completed
active ──── 6+ failures ──────► blocked (Bug Council triggered)
blocked ─── Bug Council fix ──► active (re-opened)
```

## Acceptance Criteria Format

The `criteria` field is a JSON array of strings. Each string is one independently verifiable acceptance criterion.

**Good criteria (testable, specific):**
```json
[
  "POST /api/login returns 200 with JWT token for valid credentials",
  "POST /api/login returns 401 for invalid credentials",
  "Login attempts are rate-limited to 5 per IP per minute"
]
```

**Poor criteria (vague, untestable):**
```json
[
  "Authentication works",
  "Security is good"
]
```

## Querying Contracts

```bash
DB=".dev-forge/dev-forge.db"

# All active contracts
sqlite3 "$DB" "SELECT id, task, team_lead FROM contracts WHERE status='active'"

# Contracts for a specific team
sqlite3 "$DB" "SELECT id, task, status FROM contracts WHERE team_lead='team-alpha-lead'"

# Completed contracts with notes
sqlite3 "$DB" "SELECT id, task, completed_at, notes FROM contracts WHERE status='completed' ORDER BY completed_at DESC"
```
